# 프로젝트 명세서 — AI 법률 블로그 편집기

> 최초 작성: 2026-06-25

---

## 1. 프로젝트 목표

네이버 블로그에 게시할 법률 원고를 자동으로 꾸며주는 도구.

변호사별 색상 스타일을 유지하면서, 원고를 입력하면 강조·소제목·키워드가 자동으로 적용된 붙여넣기 가능한 HTML을 즉시 생성한다.

장기 목표는 Claude API를 연결해 단순 스타일 적용을 넘어, 원고 품질 자체를 AI가 보완하는 편집기로 발전시키는 것이다.

---

## 2. 전체 기능 목록

### 현재 구현된 기능

| 기능 | 설명 |
|------|------|
| 원고 입력 | 자유 텍스트 입력, 소제목 자동 인식 |
| 변호사 프리셋 선택 | 10명 변호사별 색상 테마 |
| 키워드 자동 삽입 | 메인/서브 키워드가 없을 때 문장으로 삽입 |
| 소제목 인식 | `1. 2. 3.` 번호 형식을 소제목으로 파싱 |
| 강조 문장 자동 선택 | 섹션마다 2~3개 문장을 중요도 기준으로 강조 |
| HTML 생성 | 네이버 블로그 인라인 스타일 HTML 출력 |
| 미리보기 | 생성된 HTML 실시간 렌더링 |
| 복사 | 클립보드 복사 또는 전체 선택 |

### 향후 구현할 기능 (Claude API 연동 이후)

| 기능 | 설명 |
|------|------|
| AI 강조 문장 선택 | 단순 점수 기반 대신 Claude가 핵심 문장 판단 |
| AI 소제목 생성 | 원고 흐름을 읽고 소제목이 없을 때 자동 생성 |
| 원고 품질 개선 | 어색한 문장, 반복 표현 자동 교정 제안 |
| 키워드 자연 삽입 | 키워드를 억지로 끼워넣지 않고 문맥에 맞게 녹임 |
| 글 길이 보완 | 너무 짧은 원고에 내용 보강 제안 |
| 법률 용어 검수 | 잘못된 법률 표현 감지 및 교정 |

## 3. 핵심 규칙 (Core Rules)

> 이 규칙은 기술 명세보다 우선한다. 구현 방식이 바뀌어도 이 원칙은 유지된다.

### AI 편집 핵심 원칙

이 프로젝트의 목표는 단순한 HTML 꾸미기 프로그램이 아니다.

**사람이 직접 편집한 것처럼 자연스럽게 편집하는 AI 법률 블로그 편집기**를 만드는 것이 목적이다.

AI는 항상 원고 전체를 먼저 읽고 내용을 이해한 뒤 편집을 시작해야 한다.

편집의 목적은 내용을 바꾸는 것이 아니라 가독성을 높이고 사람이 직접 편집한 것처럼 만드는 것이다.

> **AI가 편집했다는 느낌이 들면 실패다.**

---

### Rule 1. 원고 분석

AI는 편집을 시작하기 전에 반드시 원고 전체를 읽는다.

아래 내용을 먼저 분석한다.

- 글의 주제
- 문단 흐름
- 핵심 내용
- 중요한 문장
- 문단별 역할
- 독자가 반드시 알아야 하는 내용

이 분석이 끝난 후에만 편집을 시작한다.

---

### Rule 2. 소제목 생성 규칙

#### 출력 형식

소제목은 반드시 아래 형식으로 출력한다.

```
1. 소제목

(본문)

2. 소제목

(본문)

3. 소제목

(본문)

4. 소제목

(본문)
```

반드시 **숫자 형식(1. 2. 3. 4.)** 을 사용한다.

`①`, `②`, `■`, `▶`, `●`, `-`, `*`, `※` 등 다른 기호는 사용하지 않는다.

#### 생성 원칙

AI는 먼저 소제목을 만드는 것이 아니다.

반드시 원고 전체를 끝까지 읽고 내용을 이해한 후, 글의 흐름을 가장 자연스럽게 **4개의 구간**으로 나눈다. 그 이후 각 구간의 핵심 내용을 가장 잘 설명하는 소제목을 생성한다.

> **'소제목을 먼저 만들고 문단을 맞추는 방식'이 아니라,**
> **'문단의 흐름을 먼저 이해하고 그에 맞는 제목을 붙이는 방식'으로 작업한다.**

소제목만 순서대로 읽어도 글 전체의 흐름을 이해할 수 있어야 한다.

**원고의 맨 위에는 절대로 소제목을 넣지 않는다.** 첫 문장은 항상 본문부터 시작하며, 첫 번째 소제목은 본문이 어느 정도 진행된 후부터 들어간다.

**예시**

```
업무상횡령죄가 성립하는 기준부터 살펴봐야 합니다
금액에 따라 적용 법률이 달라질 수 있습니다
실제 수사에서는 이런 부분을 중요하게 봅니다
초기 대응이 결과를 크게 바꿀 수 있습니다
```

#### 작성 스타일

소제목은 항상 같은 방식으로 작성하면 안 된다.

아래처럼 동일한 패턴을 반복해서는 안 된다.

- 전부 질문형
- 전부 `~합니다`
- 전부 `~방법`
- 전부 `~기준`
- 전부 `~체크`
- 전부 `~알아보기`

원고 내용을 바탕으로 아래 유형들을 자연스럽게 섞어 사용한다.

| 유형 | 예시 |
|------|------|
| 질문형 | `이런 경우에도 처벌받을 수 있을까요?` |
| 설명형 | `형량은 이렇게 결정됩니다` |
| 결론형 | `결국 핵심은 고의성 입증입니다` |
| 체크포인트 | `이것만은 반드시 확인하세요` |
| 주의사항 | `이 시점을 놓치면 돌이킬 수 없습니다` |
| 방법 안내형 | `초기 대응은 이렇게 준비합니다` |
| 비교형 | `민사와 형사, 무엇이 다를까요` |
| 핵심 정리형 | `실무에서 가장 자주 보는 유형입니다` |

소제목은 AI가 만든 느낌이 아니라 사람이 직접 고민해서 붙인 제목처럼 자연스러워야 한다.

#### 가장 중요한 원칙

소제목의 목적은 꾸미기가 아니라 **독자의 이해를 돕는 것**이다.

- 화려한 제목보다 본문 내용을 가장 정확하게 설명하는 제목을 우선한다.
- 클릭을 유도하기 위한 과장 표현이나 자극적인 표현은 사용하지 않는다.
- 기존 원고에 소제목이 없는 경우에는 문맥을 이해하여 새롭게 생성한다.
- 기존 원고에 소제목이 있는 경우에도 **그대로 복사하지 않고**, 참고만 하여 새롭게 작성한다.

---

### Rule 3. 강조 문장 선택 규칙

AI는 내용을 먼저 이해한 뒤 강조를 선택한다. **절대로 규칙적으로 강조하지 않는다.**

아래 방식은 사용하지 않는다.

- 매 문단마다 2개
- 첫 문장만
- 마지막 문장만

내용을 읽고 정말 중요한 부분만 강조한다.

- 소제목 하나당 약 **2~4개** 강조 권장
- 원고 전체 기준 약 **8~12개** 강조 권장

#### 강조 기준

- 처벌 / 벌금 / 형량
- 법률 기준
- 핵심 결론
- 예외 사항
- 대응 방법
- 실무상 중요한 부분
- 독자가 반드시 알아야 하는 내용

#### 가장 중요한 규칙 — 의미 단위 강조

강조는 **문장이 아니라 의미 단위**로 적용한다.

문장 전체를 강조하지 않는다. 핵심 단어 또는 핵심 구절만 강조한다.

사람이 직접 편집한 것처럼 자연스러워야 한다.

---

### Rule 4. 꾸밈효과 규칙

꾸밈효과 역시 규칙적으로 적용하면 안 된다. 모든 문단에 같은 개수의 꾸밈효과를 적용하면 안 된다.

내용을 읽고 중요한 부분에만 적용하며, 사람이 편집한 것처럼 자연스럽게 배치한다.

#### 기본 꾸밈효과 종류

- 글자색
- 글자 배경색
- 굵게
- 밑줄

#### 변호사 프리셋에 따른 차이

모든 변호사가 같은 스타일을 사용하지 않는다. 프리셋에 따라 아래 항목이 달라진다.

- 글씨 굵기 사용 여부
- 글자색
- 배경색
- 밑줄 사용 여부
- 강조 스타일 전체

기본값은 굵게 처리이지만, 변호사 프리셋에서 해제되어 있으면 굵게 처리하지 않는다.

**밑줄은 문장 전체를 긋지 않는다.** 핵심 단어 또는 핵심 구절에만 적용한다.

---

### Rule 5. 따옴표 처리

`""` 또는 `''` 안에 있는 문장은 자동으로 인식한다.

따옴표 안의 내용은 변호사 프리셋에서 지정한 **글자색**을 적용한다.

---

### Rule 6. 키워드 삽입 규칙

사용자는 메인키워드, 서브키워드, 삽입 횟수를 입력할 수 있다.

**기본값**

| 키워드 | 삽입 횟수 |
|--------|-----------|
| 메인키워드 | 7회 |
| 서브키워드 | 5회 |

#### 가장 중요한 규칙 — 자연스러운 삽입

키워드를 넣기 위해 **새로운 문장을 절대로 만들지 않는다.**

없는 내용을 추가하지 않는다. 기존 문장을 자연스럽게 수정하거나 표현을 조금 변경하는 수준에서만 삽입한다.

**글의 의미는 절대 변경하지 않는다.** 사람이 직접 수정한 것처럼 자연스럽게 삽입한다.

---

### Rule 7. AI 편집 금지 규칙

AI는 아래 행위를 해서는 안 된다.

| 금지 항목 |
|-----------|
| 없는 사실을 추가하는 것 |
| 법률 내용을 임의로 변경하는 것 |
| 기존 원고의 의미를 변경하는 것 |
| 문장의 순서를 크게 바꾸는 것 |
| 원고를 다시 작성하는 것 |

AI의 역할은 새로운 글을 만드는 것이 아니라, **사람이 직접 편집한 것처럼 자연스럽게 편집하는 것**이다.

---

### Rule 8. Claude API 역할

Claude는 **HTML을 생성하지 않는다.** Claude는 **JSON만 반환한다.**

JSON에는 아래 정보만 포함한다.

- 소제목
- 문단 구분
- 강조 위치
- 꾸밈 정보
- 키워드 정보

최종 HTML 생성은 현재 프로젝트(`htmlBuilder.ts`)에서 담당한다.

---

### 최우선 원칙

AI는 규칙대로 기계적으로 편집하면 안 된다.

항상 아래 질문을 먼저 판단한 후 편집해야 한다.

> "사람이라면 왜 이 부분을 강조했을까?"
>
> "사람이라면 왜 이 위치에 소제목을 넣었을까?"

**최종 결과는 AI가 편집한 것이 아니라 사람이 직접 편집한 것처럼 보여야 한다.**

## 4. Claude API 역할 (기술 명세)

Claude API는 현재 규칙 기반 로직을 대체하거나 보완하는 역할을 한다.

### 연결 방식

```
[사용자 원고 입력]
       ↓
[1단계] sectionBuilder — 소제목/문장 파싱 (기존 유지)
       ↓
[2단계] Claude API 호출 (현재: 규칙 기반 로직)
       ↓  - 강조할 문장 선택
       ↓  - 소제목이 없으면 생성
       ↓  - 키워드 자연 삽입 위치 결정
       ↓
[3단계] htmlBuilder — HTML 생성 (기존 유지)
```

### API를 붙이는 위치

`app/lib/analyzer.ts`의 `analyze()` 함수 내부에서 `applyDecorations()` 호출 직전에 Claude API 응답을 주입하는 구조로 설계한다.

```typescript
// 현재 (규칙 기반)
sections = applyDecorations(sections);

// 향후 (AI 기반, 폴백 포함)
const aiResult = await callClaudeApi(sections, preset);
sections = aiResult ?? applyDecorations(sections); // API 실패 시 기존 로직으로 폴백
```

### Claude에게 보낼 정보

- 원고 전문
- 소제목 목록
- 문장 목록 (섹션별)
- 변호사 이름 (문체 참고용)
- 메인/서브 키워드

### Claude에게 받을 정보

- 섹션별 강조할 문장과 강조 유형 (`text` / `bg` / `underline`)
- 소제목 생성 결과 (요청 시)
- 키워드 삽입 위치와 문장 (요청 시)

## 5. HTML 생성 역할

### 원칙

- 네이버 블로그 편집기의 HTML 모드에 직접 붙여넣을 수 있어야 한다.
- 외부 CSS 클래스 없이 인라인 스타일만 사용한다.
- 모든 텍스트는 `<p>` 태그로 감싼다.

### 출력 구조

```html
<!-- 소제목 -->
<p style="font-size:17px;font-weight:bold;margin:24px 0 10px;">1. 소제목</p>

<!-- 강조 문장 (글자색) -->
<p style="line-height:1.9;margin-bottom:16px;">
  <span style="font-weight:bold;color:#007433;">강조할 문장입니다.</span>
</p>

<!-- 강조 문장 (배경색) -->
<p style="line-height:1.9;margin-bottom:16px;">
  <span style="font-weight:bold;background-color:#e3fdc8;">강조할 문장입니다.</span>
</p>

<!-- 강조 문장 (밑줄) -->
<p style="line-height:1.9;margin-bottom:16px;">
  <span style="font-weight:bold;text-decoration:underline;">강조할 문장입니다.</span>
</p>

<!-- 일반 문장 -->
<p style="line-height:1.9;margin-bottom:16px;">일반 문장입니다.</p>

<!-- 섹션 구분 빈 줄 -->
<p style="line-height:1.9;margin-bottom:16px;">&nbsp;</p>
```

### 강조 유형별 스타일

| 유형 | 적용 스타일 |
|------|-------------|
| `text` (글자색) | `color: {textColor}` + 프리셋에 따라 `text-decoration:underline` 추가 |
| `bg` (배경색) | `background-color: {bgColor}` + darkBg 프리셋이면 `color:#ffffff` 추가 |
| `underline` (밑줄) | `text-decoration:underline` |
| 공통 | 모든 강조에 `font-weight:bold` 적용 |

## 6. 변호사별 프리셋 구조

### 데이터 구조

```typescript
interface Preset {
  name: string;            // 변호사/의뢰인 이름
  textColor: string;       // 글자색 (hex)
  bgColor: string;         // 배경색 (hex)
  textWithUnderline: boolean; // 글자색 적용 시 밑줄도 함께
  darkBg: boolean;         // 배경이 진할 때 글씨를 흰색으로
}
```

### 현재 등록된 프리셋 (10개)

| 이름 | 글자색 | 배경색 | 특이사항 |
|------|--------|--------|----------|
| 일송/이별전문/이혼 | #007433 (초록) | #e3fdc8 | - |
| 더윌가사 | #ac9a00 (골드) | #fff8b2 | - |

## 7. 키워드 자동 삽입 구조

### 동작 원칙

1. 원고 전체에 해당 키워드가 이미 포함되어 있으면 삽입하지 않는다.
2. 키워드가 없을 때만 삽입 문장을 생성한다.
3. 메인키워드와 서브키워드를 독립적으로 처리한다.

### 삽입 위치

| 키워드 종류 | 삽입 위치 | 삽입 문장 형식 |
|-------------|-----------|----------------|
| 메인키워드 | 첫 번째 섹션 맨 앞 | `오늘은 {키워드}에 대해 함께 살펴보겠습니다.` |
| 서브키워드 | 중간 섹션 맨 뒤 | `{키워드}도 이와 함께 알아두면 도움이 됩니다.` |

### 향후 개선 방향 (Claude API 연동 시)

- 현재: 고정 문장 템플릿으로 삽입
- 향후: 원고 문맥을 읽고 키워드를 자연스럽게 녹인 문장 생성
- 삽입 위치도 문맥상 가장 어울리는 곳을 AI가 판단

## 8. 소제목 처리 규칙 (현재 구현)

### 현재 인식 방식

`1.`, `2.`, `3.` 등 숫자+마침표로 시작하는 줄을 소제목으로 인식한다.

```
1. 이혼 소송의 절차      ← 소제목
이혼 소송은 ...         ← 본문
```

### 소제목 HTML 스타일

```html
<p style="font-size:17px;font-weight:bold;margin:24px 0 10px;">1. 소제목</p>
```

### 향후 개선 방향 (Claude API 연동 시)

| 상황 | 처리 방식 |
|------|-----------|
| 소제목이 있는 경우 | 기존 인식 방식 유지 |
| 소제목이 없는 경우 | Claude가 원고를 읽고 적절한 소제목 생성 |
| 소제목이 어색한 경우 | Claude가 개선안 제안 (선택 적용) |

## 9. 강조 문장 선택 규칙 (현재 구현)

### 현재 방식 (규칙 기반)

#### 섹션당 강조 개수

섹션 인덱스에 따라 2개, 3개를 교대로 적용한다.

```
섹션 0: 3개, 섹션 1: 2개, 섹션 2: 3개, 섹션 3: 2개, ...
```

#### 강조 유형 배분 패턴

섹션마다 고정 패턴으로 강조 유형을 배분한다.

```
섹션 0: [bg, underline], 섹션 1: [underline, bg], 섹션 2: [bg], ...
```

큰따옴표(`"`) 포함 문장이 있으면 `text`(글자색)를 우선 배정하고 남은 슬롯을 패턴으로 채운다.

#### 중요도 점수 산정

규칙 기반 점수로 강조 우선순위를 결정한다.

| 조건 | 가산점 |
|------|--------|
| 문장 길이 | 길이 × 0.6 |
| 첫 번째 문장 | +20점 |
| 마지막 문장 | +15점 |
| 숫자 포함 | +10점 |
| 강조 키워드 포함 (`반드시`, `꼭`, `중요`, `핵심`, `특히`, `가장`, `최고`, `최적`, `필수`) | +12점 |

### 향후 방식 (Claude API 연동 시)

Claude에게 섹션별 문장 목록을 전달하고, 다음을 요청한다.

- 독자에게 가장 중요한 문장 2~3개 선택
- 선택 이유와 적합한 강조 유형 (`text` / `bg` / `underline`) 판단
- 큰따옴표 문장의 경우 글자색 우선 적용 (현재 규칙 유지)

규칙 기반 결과는 API 실패 시 폴백으로 유지한다.

app/
